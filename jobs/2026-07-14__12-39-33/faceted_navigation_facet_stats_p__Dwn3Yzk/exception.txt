# Faceted Navigation Backend with Numeric Facet Statistics (Typesense)

## Background
You are building the backend for a product catalog's faceted navigation UI, powered by Typesense (a standalone native Linux binary is available in this environment). Shoppers refine results by selecting brands, categories, and tags, and by dragging a price slider. For every facet, the UI must show how many products would match, following the same multi-select semantics used by real e-commerce sites.

A Typesense server binary is installed at `/usr/local/bin/typesense-server`. A product dataset is provided at `/home/user/facet-nav/data/products.jsonl` (JSONLines, one product per line).

## Requirements
- Run Typesense on port 8108 and index every product from the provided dataset into a collection named `products`.
- Enable faceting on `brand`, `category`, `tags` (string array) and `price`.
- Implement a rerunnable CLI that accepts a search/navigation request and returns facet counts, numeric price statistics, and (optionally) facet-value autocomplete matches.
- Facet counts must correctly combine the active filters with the facet aggregation.

## Implementation Hints
- Use the Typesense Python SDK to create the schema, import the dataset, and run searches.
- The dataset fields are: `product_name` (string), `brand` (string), `category` (string), `tags` (string array), `price` (float), `rating` (float).
- Numeric price statistics (min/max/avg/sum) come from Typesense's facet stats when you facet on a numeric field; make sure the counts and statistics are exact rather than sampled or approximate.
- Multi-select (disjunctive) faceting is the core challenge: the counts returned for a given facet field must reflect all currently active filters EXCEPT any filter applied to that same field, so a shopper who has already selected "Apple" still sees the counts of the other brands they could switch to. In contrast, `found` and `price_stats` must reflect ALL active filters combined.
- `facet_query` performs a case-insensitive prefix match over a field's facet values; use it to power the facet search box.
- `max_facet_values` caps how many facet values are returned for each field.

Project path: /home/user/facet-nav

### Setup / indexing entry point
Provide an executable indexing step at `/home/user/facet-nav/index_data.py` that connects to Typesense on `localhost:8108`, (re)creates the `products` collection with the appropriate schema, and imports all products from `data/products.jsonl`. Running it repeatedly must leave the collection in a clean, fully-indexed state.

### Query CLI
- Command: `python3 /home/user/facet-nav/facet_nav.py`
- The command reads a single JSON object from **stdin** and writes a single JSON object to **stdout**.
- Connect using API key `xyz`, host `localhost`, port `8108`, protocol `http`.

Request object (stdin):
```json
{
  "q": "*",
  "filters": {
    "brand": ["Apple"],
    "category": ["Laptops"],
    "tags": ["premium"],
    "price": { "min": 500, "max": 2000 }
  },
  "facet_query": { "field": "brand", "prefix": "sa" },
  "max_facet_values": 10
}
```
- `q` is optional and defaults to `"*"` (match all). When `q` is not `"*"`, search against `product_name`.
- `filters` is optional. Any of `brand`, `category`, `tags` may be provided as a list of selected values (a document matches a field's filter if it equals any selected value). `price` is optional and given as an object with numeric `min` and/or `max` (inclusive bounds).
- `facet_query` is optional.
- `max_facet_values` is optional and defaults to `10`.

Response object (stdout):
```json
{
  "found": 6,
  "facets": {
    "brand": [ { "value": "Apple", "count": 2 } ],
    "category": [ { "value": "Laptops", "count": 6 } ],
    "tags": [ { "value": "premium", "count": 4 } ]
  },
  "price_stats": { "min": 599.0, "max": 1999.0, "avg": 1199.0, "sum": 7194.0 },
  "facet_query_matches": [ { "value": "Samsung", "count": 3 } ]
}
```
- `found` is the number of products matching `q` and ALL active filters.
- `facets` must include the keys `brand`, `category`, and `tags`; each maps to a list of `{ "value": <string>, "count": <integer> }` objects sorted by `count` in descending order, following the disjunctive semantics described above.
- `price_stats` must contain the numeric keys `min`, `max`, `avg`, and `sum`, computed over the products matching `q` and ALL active filters, each rounded to 2 decimal places. If no products match, use `0` for every statistic.
- `facet_query_matches` must be present only when `facet_query` is supplied; it lists the `{ "value", "count" }` matches (case-insensitive prefix) for the requested field under the current active filters, sorted by `count` descending.
