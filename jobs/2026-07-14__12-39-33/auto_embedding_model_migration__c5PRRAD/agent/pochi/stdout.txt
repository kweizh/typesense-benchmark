## What happened

### The non-obvious constraint
Typesense stores raw field data inside each document record even after a field is dropped from the schema. When you try to re-add a vector field with a different `num_dim`, Typesense re-validates every stored document against the new field definition — and rejects the operation if any document's stored value has the wrong number of dimensions. A simple drop-then-add at a different dimensionality therefore fails.

### Migration steps (in `/home/user/migration/migrate.py`)

| Step | Action | Why |
|------|--------|-----|
| 1 | `PATCH /collections/notes` → drop `content_embedding` from the schema | Removes the schema-level vector index |
| 2 | Export all 8 documents, strip the `content_embedding` key, re-upsert | Clears the stale 4-dim raw data from stored document records |
| 3 | `PATCH` → add `content_embedding` back with `num_dim=8, optional=true` | `optional=true` allows existing docs (which have no value yet) to pass validation |
| 4 | Bulk-import `new_vectors.jsonl` with `action=update` | Writes the 8-dim vectors into every document and builds the HNSW index |
| 5 | Drop + re-add the field with `optional=false` | Enforces the required field on all future documents |
| 6 | Full document verification — every doc checked for correct dims and unchanged `title`/`content`/`category` | Confirms data integrity |
| 7 | Vector search smoke test (query nearest to doc 1's new vector) | Confirms the HNSW index is functional |
