## Files

- **`setup.py`** — idempotently (re)creates the three collections (`products`, `articles`, `users`) with schemas matching the JSONL fields, deletes and recreates them on every run, bulk-imports each JSONL via `/collections/{name}/documents/import`, and verifies each collection reports the expected document count.
- **`search.py`** — takes `--query "<q>"`, builds one federated `/multi_search` POST whose body has three sub-queries (one per collection), and emits a single JSON line on stdout, exiting 0.

## Key design decisions

- **Single federated `multi_search`** — verified via instrumentation: exactly **1 HTTP request** with 3 sub-queries. Each sub-query carries its own `collection`, `q`, and `query_by` (`product_name` for products, `title,body` for articles, `username,full_name` for users).
- **Per-query vs. common params** — per-query params live inside each individual search object; no shared common params are needed beyond the `searches` array.
- **Output shape** — top-level `{"query": str, "results": {"products": …, "articles": …, "users": …}}`. For a successful sub-query: `{"found": int, "hits": [<doc>, …]}` where each hit is unwrapped to the actual matched document object (so `hits[i].id` works). For a failing sub-query: `{"error": str}`.
- **Resilience** — each result slot is inspected independently. If a slot carries `error`/`code`, that collection becomes `{"error": "..."}` and the other slots' results are preserved. If the whole HTTP request fails (auth/network), every collection is reported as an error rather than crashing. All three keys are always present.
