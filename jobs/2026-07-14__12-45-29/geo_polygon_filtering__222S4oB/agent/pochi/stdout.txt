# Typesense Geo-Polygon Fencing for Delivery Hubs ## Background A logistics company runs a network of delivery hubs, each tagged with a GPS coordinate and an operational status. ## Files

**`/home/user/geo-search/start.sh`**
- Launches `/usr/local/bin/typesense-server` on `localhost:8108` using the API key from `$TYPESENSE_API_KEY`, with a local data dir, log file, and pid file under `/home/user/geo-search/`.
- Polls `http://localhost:8108/health` until `{"ok":true}`.
- Idempotent: if a healthy server is already listening, it simply returns 0 in milliseconds (verified).
- Detached via `nohup ... &`, so the server keeps running between seed and search calls.

**`/home/user/geo-search/seed.py`**
- Drops and recreates the `hubs` collection, declaring `name` (string), `status` (string, `facet: True`), and `location` (`geopoint`).
- Loads all 10 hubs from the spec, storing `location` as `[latitude, longitude]` in that exact order.
- Uses `action=create` against the freshly-empty collection (the Python client v0.21 rejects the deprecated `replace` action).
- Polls `num_documents` until all 10 are indexed before exiting, so re-running is safe.

**`/home/user/geo-search/search.py`**
- Parses `--polygon "lat1,lng1,lat2,lng2,..."` (the comma-separated lat/lng string) into a list of `(lat, lng)` tuples, validating that the count is even and ≥ 6 numbers.
- Builds a Typesense filter clause of the form `location:(lat,lng,lat,lng,...)` — true point-in-polygon containment, no radius approximation.
- When `--exclude-status <status>` is given, ANDs in `status:!=<value>` (not-equal).
- Pages through results with `per_page=100` to guarantee no silent truncation.
- Sorts the returned ids ascending-lexicographically and prints exactly one JSON object: `{"hub_ids": [...]}` (or `{"hub_ids": []}` when nothing matches).
